# 系統分析與設計：課程說明與規定

## 課程名稱

系統分析與設計

## 課程目標

本課程以一個小組專案貫穿整個暑修學期。課程重點不是只背系統開發生命週期，也不是只畫統一塑模語言圖（UML diagram），而是能夠從真實或半真實情境出發，逐步完成問題定義、需求分析、系統分析、系統設計、原型展示與測試案例。

完成課程後，應能：

1. 定義資訊系統問題，辨識利害關係人（stakeholders）、系統邊界與專案範圍。
2. 撰寫專案章程（Project Charter）與問題陳述（problem statement）。
3. 蒐集、整理與驗收需求，完成需求文件包（Requirements Package）。
4. 建立分析模型（Analysis Models），包含使用案例（Use Case）、資料流程圖（DFD）與實體關係圖（ERD）。
5. 撰寫軟體需求規格書（SRS, Software Requirements Specification）與軟體設計文件（SDD, Software Design Document）。
6. 製作原型（prototype），並以使用者任務進行可用性測試（usability test）。
7. 建立測試案例（test cases）與期末專案交付包（Final Package）。
8. 使用 GitHub 文件與版本管理平台（GitHub）整理小組文件、模型、原型連結與繳交紀錄。

## 使用工具

本課程主要使用：

- GitHub 文件與版本管理平台（GitHub）
- GitHub 儲存庫（GitHub repository）：每組一個小組專案儲存庫
- Markdown 標記語言（Markdown）：撰寫 `README.md`、需求文件與專案紀錄
- draw.io 線上繪圖工具（draw.io）、Figma 介面設計工具（Figma）、Canva 設計工具（Canva）、PowerPoint 簡報工具（PowerPoint）或其他可產出模型與原型的工具
- 生成式 AI 工具（generative AI tools）：可輔助發想、整理與檢查，但不可取代小組判斷

GitHub 儲存庫（GitHub repository）建立時請勾選：

```text
Add a README file
```

建議小組儲存庫名稱：

```text
TKU_114-3_SAD_Group01
```

請將 `Group01` 改成自己的組別，例如：

```text
TKU_114-3_SAD_Group03
```

## GitHub 使用規定

本課程使用 GitHub 管理小組專案文件，不要求每組一定要寫出完整可部署程式。GitHub 的用途是保存版本、整理交付物、留下修改紀錄，並讓老師能檢查每週進度。

每組只需要一個小組 GitHub 儲存庫（GitHub repository）。組員應共同維護同一個儲存庫，不要每個人各自交一份不同版本。

建議資料夾結構：

```text
TKU_114-3_SAD_Group01/
├── README.md
├── 0706_Project_Charter/
│   └── project_charter_v1.md
├── 0707_Stakeholder_Feasibility/
│   └── stakeholder_feasibility_v1.md
├── 0713_Requirements/
│   └── requirements_package_v1.md
├── 0722_SRS_Analysis_Models/
│   ├── srs_v1.md
│   └── analysis_models.md
├── 0729_SDD_Prototype/
│   ├── sdd_v1.md
│   └── prototype_links.md
├── 0803_Test_Risk/
│   └── test_cases_risk_review.md
└── 0805_Final_Package/
    ├── srs_final.md
    ├── sdd_final.md
    ├── final_presentation_link.md
    └── ai_use_log.md
```

每次完成課堂產出後，請至少做一次提交（commit）並推送（push）到 GitHub。

建議提交訊息（commit message）：

```text
Add 0706 project charter
Update requirements package
Revise SRS based on feedback
Add prototype links and usability notes
Add final package
```

## 教材與用書

本課程不用購買教科書。

上課請依照講義、課堂工作紙與小組專案進度操作。每次課程會搭配當日任務，最後整合成期末專案交付包（Final Package）。

## 成績配分

| 項目 | 比例 |
|---|---:|
| 平時專案進度 | 40% |
| 期中專案 | 30% |
| 期末專案 | 30% |
| 出席與課堂參與 | 額外加分最多 10 分 |

說明：

- 平時專案進度 40%：以每日 / 每週小組產出、GitHub 紀錄、課堂討論與修正情況為主。
- 期中專案 30%：以軟體需求規格書（SRS）、分析模型（Analysis Models）與期中專案檢核為主。
- 期末專案 30%：以期末專案交付包（Final Package）、原型展示（prototype demo）與期末簡報為主。
- 出席與課堂參與為額外加分，最多加 10 分；總成績上限依學校成績系統規定處理。

## 作業與交付物規定

本課程作業以小組專案文件與原型為主。每次作業請放在小組 GitHub 儲存庫（GitHub repository）的指定資料夾中，並到 iClass 教學平台（iClass）繳交該資料夾連結。

主要交付物如下：

| 週次 | 交付物 | 內容 |
|---|---|---|
| 第 1 週 | 專案章程（Project Charter） | 題目、目標使用者、問題陳述、範圍內 / 範圍外、核心任務 |
| 第 1 週 | 利害關係人與可行性簡報（Stakeholder & Feasibility Brief） | 利害關係人地圖、可行性分析、初步風險 |
| 第 2 週 | 需求文件包（Requirements Package） | 訪談腳本、需求來源、需求清單、使用者故事、驗收條件 |
| 第 3 週 | 軟體需求規格書（SRS）與分析模型（Analysis Models） | 使用案例、流程模型、資料流程圖、實體關係圖 |
| 第 4 週 | 軟體設計文件（SDD）與原型（Prototype） | 架構、類別圖、循序圖、介面流程、原型連結與測試紀錄 |
| 第 5 週 | 測試案例與風險檢查（Test Cases & Risk Review） | 測試案例、權限檢查、風險與安全檢查 |
| 第 5 週 | 期末專案交付包（Final Package） | SRS final、SDD final、prototype、test cases、AI 使用紀錄、期末簡報 |

作業原則：

1. 課堂作業期限依 iClass 教學平台（iClass）公告為準。
2. 作業以 GitHub 資料夾連結、檔案內容與提交紀錄（commit history）為主要認定依據。
3. 只上傳截圖、只貼簡報、只貼聊天紀錄，原則上不算完整作業。
4. 每次作業需能看出小組本次新增或修正了什麼內容。
5. 文件可以用 Markdown 標記語言（Markdown）撰寫；圖檔、PDF、Figma 連結或其他工具連結可以放在同一資料夾的說明檔中。

## iClass 與 GitHub 繳交規定

每次作業完成後，請確認 GitHub 儲存庫（GitHub repository）上看得到指定資料夾，並到 iClass 教學平台（iClass）繳交該資料夾連結。

繳交格式：

```text
https://github.com/你的帳號/TKU_114-3_SAD_GroupXX/tree/main/日期或交付物資料夾
```

範例：

```text
https://github.com/pei/TKU_114-3_SAD_Group03/tree/main/0706_Project_Charter
```

不要只貼儲存庫首頁，例如：

```text
https://github.com/pei/TKU_114-3_SAD_Group03
```

繳交前請檢查：

1. 連結打開後看得到指定資料夾。
2. 資料夾中看得到當次作業檔案。
3. `README.md` 或資料夾內說明檔有清楚標示本次交付內容。
4. 若使用外部工具，例如 Figma 介面設計工具（Figma）或 draw.io 線上繪圖工具（draw.io），連結權限要能開啟。
5. GitHub 上看得到最新提交（commit）與推送（push）紀錄。

## 上課方式

每次課程通常包含：

1. 概念說明。
2. 案例分析。
3. 小組實作。
4. 文件或模型修正。
5. GitHub 提交（commit）與推送（push）。
6. iClass 教學平台（iClass）繳交。

本課重點是把每次課堂產出累積成完整期末專案，不是每週做互不相關的小作業。

## 點名與出席規定

出席與課堂參與為額外加分，最多加 10 分。

點名方式依課堂公告辦理。若當日有特殊情況，以課堂公告為準。

出席規定：

1. 暑修課程進度密集，缺席會直接影響小組產出。
2. 課堂工作坊與小組討論需在課堂中完成，原則上不接受只在課後補文字。
3. 若因特殊原因無法到課，請依學校規定辦理，並主動與小組協調負責工作。
4. 小組成員若長期未參與，期末可依同儕回饋調整個人成績。

## AI 工具使用規定

可以使用生成式 AI 工具（generative AI tools）協助：

- 發想專案情境。
- 檢查問題陳述是否具體。
- 整理訪談摘要或需求草稿。
- 檢查需求、模型與文件是否前後矛盾。
- 協助產生測試案例草稿。
- 協助改善文件文字清楚度。

不可以做：

- 直接把生成式 AI（generative AI）產生的完整文件當成小組成果。
- 捏造不存在的訪談、使用者回饋或資料來源。
- 讓 AI 替小組決定需求優先順序、系統邊界或設計取捨。
- 繳交小組成員無法解釋的內容。
- 未揭露 AI 使用情況。

若使用生成式 AI（generative AI），每次作業需附 AI 使用紀錄（AI use log），至少包含：

1. 使用目的。
2. 提示詞摘要（prompt summary）。
3. AI 產出重點。
4. 小組採用哪些內容。
5. 小組不採用哪些內容。
6. 人工修正內容。

## 學術誠信與小組分工

可以討論觀念、文件架構、模型畫法與工具操作，但交付內容必須是小組依據自己題目完成的成果。

以下情況可能被視為不當繳交：

1. 多組繳交高度相同的題目、需求、模型或文件。
2. 直接複製網路範本、同學文件或 AI 產出，未進行小組情境化修正。
3. 捏造訪談紀錄、使用者回饋、測試結果或資料來源。
4. 只有少數人完成全部內容，其他成員無法說明小組專案。
5. GitHub 提交紀錄長期只由單一成員完成，且小組無法說明分工。

## 小組分工建議

每組可以依能力分工，但所有組員都必須理解整體專案。

建議角色：

1. 專案整合：負責維護 README、交付物清單與版本狀態。
2. 需求分析：負責訪談、需求清單、使用者故事與驗收條件。
3. 模型設計：負責使用案例、流程、DFD、ERD、類別圖與循序圖。
4. 原型設計：負責介面流程、prototype 與 usability notes。
5. 測試與品質：負責測試案例、風險檢查、AI 使用紀錄與文件一致性。

角色可以重疊，也可以每週輪換。期末每位成員都需要能說明自己負責的內容。

## 缺席與遲交

作業期限不因缺席自動延長。若有特殊原因，請依學校規定辦理，並主動說明狀況。

遲交處理依 iClass 教學平台（iClass）公告與課堂規定為準。若小組因成員缺席影響進度，請及早調整分工，不要等到繳交前才處理。

## 上課準備

每次上課前請確認：

1. 可以登入 GitHub。
2. 可以開啟小組 GitHub 儲存庫（GitHub repository）。
3. 可以編輯小組共用文件。
4. 上一份作業已經推送（push）到 GitHub。
5. iClass 教學平台（iClass）上的作業連結已確認可開啟。
6. 若使用 Figma、draw.io、Canva 或其他工具，連結權限已開放。
7. 小組知道本次課程要完成哪一個交付物。

## 補充建議

建議養成以下習慣：

1. 每次修改文件後都做提交（commit），不要到期末才一次上傳。
2. 提交訊息（commit message）要能看出本次修改內容。
3. 每個資料夾都放一個簡短說明檔，說明該資料夾內容。
4. 圖檔、原型連結與文件要互相對應，不要各做各的。
5. 每週檢查需求、模型、設計與測試是否一致。
6. 不要在文件中使用真實個資；若需要範例資料，請使用假資料或匿名資料。
