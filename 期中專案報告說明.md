# 系統分析與設計：期中專案成果驗證影片

## 作業名稱

期中專案：10 分鐘成果驗證影片（Midterm Project Progress Validation Video）

## 課程名稱

系統分析與設計

## 作業目的

期中專案用來呈現小組目前已完成的系統分析與實作成果。影片不需要逐份朗讀文件，重點是讓觀看者理解：

1. 小組正在解決什麼問題。
2. 需求從哪裡來。
3. 分析模型如何描述角色、流程與資料。
4. 實作成果是否符合需求與驗收條件。
5. 目前有哪些限制，以及後續要如何完成期末專案。

期中成果不要求系統全部完成。尚未完成的功能可以誠實說明，不可用靜態畫面、預錄動畫或口頭描述假裝功能已經可以操作。

## 繳交方式

每組只需繳交一支 10 分鐘成果驗證影片，不需另外繳交書面報告或簡報檔。

繳交方式依 iClass 教學平台（iClass）公告為準，可繳交影片檔案或可觀看的影片連結。若使用連結，請確認：

1. 連結不需要額外申請權限。
2. 使用一般瀏覽器即可播放。
3. 影片聲音清楚，畫面文字可以閱讀。
4. 連結在截止時間後仍可正常觀看。

建議檔名：

```text
系統分析與設計_期中專案_GroupXX_專案名稱.mp4
```

## 建議影片包含內容

各組可依專案特色安排影片順序，不需要使用固定時間分段。建議包含以下內容：

### 1. 專案與問題

簡要說明：

1. 小組與專案名稱。
2. 目標使用者或主要角色。
3. 專案要解決的核心問題。
4. 本期系統範圍內與範圍外的重要項目。

### 2. 需求成果

建議呈現：

1. 需求如何蒐集或整理。
2. 至少 3 項高優先需求。
3. 需求來源或證據。
4. 需求優先順序的主要判斷。
5. 代表性的驗收條件（acceptance criteria）。

不需要逐條朗讀全部需求，應選擇最能說明系統價值與核心流程的內容。

### 3. 分析模型

依目前完成進度選擇重要模型呈現，例如：

1. 使用案例圖（Use Case Diagram）。
2. 使用案例描述（Use Case Description）。
3. 現況／目標流程（AS-IS/TO-BE process）。
4. 資料流程圖（DFD, Data Flow Diagram）。
5. 實體關係圖（ERD, Entity Relationship Diagram）。
6. 需求追溯矩陣（requirements traceability matrix）。

模型不需要逐張完整解說，但應說明它如何支持核心需求與系統實作。

### 4. 系統實作成果

影片必須實際操作目前的系統或可操作原型（operational prototype），不能只展示截圖。

建議展示：

1. 一段核心使用者流程。
2. 一個正常情境。
3. 一個替代、例外或邊界情境。
4. 操作前後的資料或狀態變化。
5. 功能對應的需求與驗收條件。
6. 使用假資料或尚未串接正式服務的限制。

實作成果不需要功能很多，但展示的流程必須真的可以操作，並且與需求及分析模型一致。

### 5. 需求、模型與實作追溯

建議選擇至少一項核心功能，說明完整追溯關係：

```text
需求來源
→ 使用者故事或功能性需求
→ 使用案例或流程模型
→ 畫面與系統行為
→ 驗收條件與測試結果
```

這一部分的重點是證明實作不是由工具隨意產生，而是根據系統分析結果完成。

### 6. 目前限制與下一步

簡要說明：

1. 目前尚未完成的功能。
2. 已知錯誤或未通過的驗收條件。
3. 尚待確認的需求或規則。
4. 期末專案預計優先完成或改善的內容。

誠實說明限制不會因「尚未全部完成」直接扣分；隱藏問題、誤稱功能完成或無法說明目前狀態，才會影響評分。

## 小組參與

每位組員都應在影片中參與說明，並能說明自己負責的需求、模型、實作、測試或整合工作。

可以使用：

1. 投影片與口頭說明。
2. GitHub 文件與版本管理平台（GitHub）畫面。
3. 模型圖與需求文件。
4. 系統操作錄影。
5. 字幕與重點標示。

不要求每位組員平均分配相同秒數，但不應只有一位組員完成全部報告。

## GitHub 與實作展示

影片中建議實際開啟小組 GitHub 儲存庫（GitHub repository），讓觀看者看到：

1. 需求與分析模型所在位置。
2. 可操作系統或原型所在位置。
3. 測試或驗收紀錄。
4. 提交紀錄（commit history）。
5. 生成式 AI 與程式碼代理使用紀錄。

本次只需繳交影片，不需另外繳交 GitHub 連結；但影片中展示的文件與實作應能正常開啟。

## 評分規準

期中專案占學期總成績 30%。影片以 100 分評分後換算。

| 評分項目 | 分數 | 評分重點 |
| --- | ---: | --- |
| 問題定義與系統範圍 | 10 | 能清楚說明核心問題、目標角色及本期範圍 |
| 需求成果 | 15 | 高優先需求有來源、價值、排序與驗收依據 |
| 分析模型 | 20 | 模型正確、可理解，並能支持需求與核心流程 |
| 系統實作展示 | 30 | 實際操作核心流程，包含正常與替代、例外或邊界情境，資料與狀態變化正確 |
| 需求、模型與實作一致性 | 15 | 能展示需求來源到模型、實作與驗收的追溯關係 |
| 影片表達與小組參與 | 10 | 內容有組織、畫面與聲音清楚、組員皆有參與且未超過規定時間 |
| 總分 | 100 | 換算為學期總成績 30% |

## 評分判斷

### 完整

1. 問題、需求、模型與實作能形成一致的說明。
2. 有實際操作，不只展示畫面或投影片。
3. 能展示正常與非正常情境。
4. 能指出目前限制與未完成項目。
5. 模型、實作與驗收可追溯。

### 部分完成

1. 能說明專案與需求，但模型或實作連結不完整。
2. 系統可操作，但只有正常流程或缺乏驗收依據。
3. 模型已完成，但與原型的對應說明不足。
4. 影片內容可理解，但小組分工與貢獻不清楚。

### 需修正

1. 主要內容都在介紹題目，沒有需求與分析證據。
2. 只播放投影片或截圖，沒有實際操作系統。
3. 模型與實作明顯不一致。
4. 宣稱功能已完成，但無法操作或沒有驗收結果。
5. 程式碼代理加入的功能無法對應需求來源。

## 生成式 AI 與程式碼代理使用

可以使用生成式 AI（generative AI）或程式碼代理（code agent）協助：

1. 整理報告結構。
2. 檢查字幕與文字。
3. 協助剪輯或產生畫面標示。
4. 根據已確認需求協助實作與修正。

不得使用工具：

1. 捏造訪談內容或需求來源。
2. 偽造尚未完成的系統操作。
3. 產生與實際專案不一致的模型或測試結果。
4. 隱藏工具產出的錯誤或範圍外功能。

影片中不需要逐條朗讀生成式 AI 使用紀錄，但應能從 GitHub 文件與版本管理平台中的紀錄看出工具產出、人工判斷與修正。

## 繳交前檢查

1. 影片長度不超過 10 分鐘。
2. 影片聲音清楚，模型與系統畫面文字可辨識。
3. 有說明問題、需求與系統範圍。
4. 有呈現重要分析模型。
5. 有實際操作核心流程。
6. 有展示正常與替代、例外或邊界情境。
7. 有說明需求、模型與實作的對應。
8. 有說明目前限制與下一步。
9. 每位組員都有參與。
10. 影片檔案或連結可正常播放。

## 常見問題

### 系統還沒有全部完成，可以繳交嗎？

可以。期中專案評量目前成果、分析品質與一致性，不要求完成期末版本。請清楚說明已完成、未完成與目前限制。

### 一定要做投影片嗎？

不用。可以直接使用 GitHub 文件、模型圖、系統畫面與口頭說明完成影片。

### 需要另外交簡報或報告文件嗎？

不用，本次只繳交 10 分鐘成果驗證影片。

### 可以剪輯影片嗎？

可以，但剪輯後仍須呈現真實操作結果，不得使用剪輯掩蓋無法操作、錯誤或未完成狀態。

### 實作成果可以使用假資料嗎？

可以。請在影片中清楚說明假資料、角色切換與未串接正式服務等限制。
