## 學習目標

完成本單元後，你應能：

1. 將需求來源表中的證據改寫為使用者故事（user story），並保留來源編號。
2. 使用「角色、需求、價值」三部分撰寫可理解的使用者故事。
3. 區分功能性需求（FR）、非功能性需求（NFR）、業務規則（business rule）、資料需求（data requirement）與限制條件（constraint）。
4. 把模糊的品質描述改寫為可觀察、可測量或可操作檢查的非功能性需求。
5. 使用前提／當／那麼格式（Given/When/Then）撰寫驗收條件。
6. 為正常流程、例外流程與邊界條件建立驗收條件。
7. 使用必須／應該／可以／本次不做方法（MoSCoW method）與價值投入矩陣（value-effort matrix）排定需求優先順序。
8. 找出需求之間的衝突、重複、遺漏、依賴與範圍錯誤。
9. 把高優先需求轉成實作待辦清單（implementation backlog）與程式碼代理實作任務書。
10. 使用程式碼代理（code agent）修正第一個可操作切片，或完成第二個可操作切片。
11. 依驗收條件進行手動驗收測試（manual acceptance test），並保存通過、失敗與修正證據。
12. 建立需求、畫面、實作與測試之間的追溯關係（traceability）。

## 問題情境

完成訪談與需求來源表後，需求仍然可能無法直接交給實作者。常見情況包括：

1. 「要有活動報名」只有功能名稱，沒有角色、條件與完成結果。
2. 「系統要好用」沒有可觀察標準，無法判定是否完成。
3. 每一項需求都被標為最高優先，等於沒有排序。
4. 程式碼代理先做了登入、資料庫、通知與管理後台，但本次只需要完成報名流程。
5. 畫面可以操作，卻不知道對應哪一筆需求、哪一項驗收條件。
6. 驗收只看「能不能開啟」，沒有測試資料錯誤、名額已滿或重複送出等例外情境。
7. 修正程式後沒有更新文件，造成需求文件與實際原型不一致。

系統分析與設計的核心不是把文件寫多，而是讓每一個系統行為都能回答：

```text
為誰做？
為什麼需要？
來源在哪裡？
系統必須做什麼？
做到什麼程度？
這次先做哪一項？
如何證明已經完成？
```

## 核心概念

### 概念 1：需求要形成可追溯的證據鏈

需求追溯（requirements traceability）是指每個需求都能向前追到來源，也能向後追到設計、實作與測試。

完整證據鏈範例：

```text
來源編號 SRC-STU-03
→ 使用者故事 US-03
→ 功能性需求 FR-05
→ 驗收條件 AC-05-01、AC-05-02
→ 實作待辦 IMP-04
→ 畫面 SCR-02
→ 手動測試 MAT-05-01、MAT-05-02
```

每個編號的用途：

| 編號前綴 | 中文名稱 | 英文名稱 | 用途 |
| --- | --- | --- | --- |
| `SRC` | 需求來源 | requirements source | 保存訪談、文件、觀察或規則來源 |
| `US` | 使用者故事 | user story | 從使用者角度描述需求與價值 |
| `FR` | 功能性需求 | functional requirement | 規定系統必須提供的行為 |
| `NFR` | 非功能性需求 | non-functional requirement | 規定品質、限制或服務水準 |
| `BR` | 業務規則 | business rule | 規定組織流程與判斷規則 |
| `DR` | 資料需求 | data requirement | 規定必須保存或使用的資料 |
| `AC` | 驗收條件 | acceptance criterion | 規定需求何時算完成 |
| `IMP` | 實作待辦 | implementation backlog item | 把需求轉成可執行的實作工作 |
| `SCR` | 畫面或操作 | screen or interaction | 對應使用者可見介面或操作 |
| `MAT` | 手動驗收測試 | manual acceptance test | 記錄實際驗收步驟與結果 |

如果原始來源改變，應沿著證據鏈檢查所有受影響的使用者故事、需求、畫面與測試，而不是只改其中一份文件。

### 概念 2：使用者故事描述角色、需求與價值

使用者故事（user story）常用格式：

```text
身為［角色］，
我想要［完成某項行為或達成某個目標］，
以便［獲得價值或解決問題］。
```

範例：

```text
身為一般學生，
我想要在活動尚有名額時完成報名，
以便保留參加資格並確認報名狀態。
```

使用者故事的三個部分：

1. 角色（role）：誰需要這項能力。
2. 需求或目標（need or goal）：角色想完成什麼。
3. 價值（value）：完成後解決什麼問題。

不完整範例：

```text
身為學生，我想要有活動報名功能。
```

問題在於「有功能」不是使用者目標，也沒有說明完成後的價值。可改為：

```text
身為學生，我想要查看符合參加資格且尚有名額的活動，以便選擇並完成報名。
```

### 概念 3：使用者故事不是完整需求規格

一則使用者故事不應承擔所有細節。它主要負責表達角色、需求與價值，細節應分別放入：

1. 功能性需求（functional requirement）：系統必須做什麼。
2. 非功能性需求（non-functional requirement）：做到什麼品質。
3. 業務規則（business rule）：什麼條件下允許或禁止。
4. 資料需求（data requirement）：要輸入、保存或顯示哪些資料。
5. 驗收條件（acceptance criterion）：怎樣算完成。

因此，使用者故事不是軟體需求規格書（SRS, Software Requirements Specification）的替代品，而是需求溝通與工作拆分的入口。

### 概念 4：用「獨立、可協商、有價值、可估算、小型、可測試」檢查使用者故事

使用者故事可用六項品質原則（INVEST principles）檢查：

| 原則 | 英文 | 檢查問題 |
| --- | --- | --- |
| 獨立 | Independent | 是否能在不依賴太多未完成項目的情況下理解或實作？ |
| 可協商 | Negotiable | 是否保留討論空間，而不是先把技術方案寫死？ |
| 有價值 | Valuable | 是否對明確角色提供可說明的價值？ |
| 可估算 | Estimable | 小組是否知道大致需要哪些工作？ |
| 小型 | Small | 是否能在一個短實作週期內完成？ |
| 可測試 | Testable | 是否能寫出清楚的驗收條件？ |

如果一則故事包含「瀏覽、報名、付款、審核、通知、報表」，通常太大，應依使用者目標或操作流程拆分。

### 概念 5：功能性需求規定系統必須做什麼

功能性需求（functional requirement, FR）描述系統必須提供的行為、計算、資料處理或回應。

建議句型：

```text
當［條件或事件］發生時，
系統應［執行可觀察的行為］，
並［產生可確認的結果］。
```

範例：

```text
FR-05：當活動尚有名額且學生符合參加資格時，系統應接受報名，建立報名紀錄，並顯示報名成功狀態。
```

功能性需求至少要能回答：

1. 誰或什麼事件觸發？
2. 系統做什麼？
3. 在什麼條件下執行？
4. 產生什麼結果？
5. 是否能透過畫面、資料或狀態觀察？

避免使用「支援」「最佳化」「智慧化」「處理」等沒有具體行為的詞，除非後面清楚寫出系統實際要做的事。

### 概念 6：非功能性需求規定系統做到什麼程度

非功能性需求（non-functional requirement, NFR）描述系統的品質屬性（quality attribute）、服務水準、限制或整體行為標準。

常見類型：

| 中文類型 | 英文名稱 | 可檢查方向 |
| --- | --- | --- |
| 效能 | performance | 回應時間、處理量、同時使用人數 |
| 可用性 | usability | 任務完成步驟、錯誤提示、易讀性 |
| 可靠性 | reliability | 失敗後資料是否保留、重試是否重複建立資料 |
| 安全性 | security | 權限、輸入檢查、敏感資料保護 |
| 隱私 | privacy | 個人資料蒐集、顯示、保存與刪除 |
| 可維護性 | maintainability | 模組分離、命名、文件與修改影響範圍 |
| 相容性 | compatibility | 支援的瀏覽器、裝置或檔案格式 |
| 可存取性 | accessibility | 鍵盤操作、文字替代、色彩對比 |

不夠好的寫法：

```text
NFR-01：系統要快速。
NFR-02：系統要安全。
NFR-03：系統要好用。
```

較好的教學練習寫法：

```text
NFR-01：在使用本課提供的假資料、一般校園網路與支援的桌面瀏覽器環境下，活動列表操作後應在 2 秒內顯示結果。

NFR-02：未具管理者角色的使用者不可從介面執行活動核准或刪除操作。

NFR-03：當必填欄位未填寫時，系統應在對應欄位附近顯示可理解的繁體中文錯誤訊息，並保留已輸入的其他內容。
```

數字不是越精確越好。數值必須能說明來源、測試環境與測量方式；若只是小組暫定，應標示為「待確認目標」，不可假裝是利害關係人已確認的正式標準。

### 概念 7：用品質屬性情境補完整非功能性需求

品質屬性情境（quality attribute scenario）可用六個欄位整理非功能性需求：

1. 來源（source）：誰或什麼產生事件。
2. 刺激（stimulus）：發生什麼情況。
3. 環境（environment）：在什麼使用條件下。
4. 對象（artifact）：哪個系統、模組或資料受到影響。
5. 回應（response）：系統應如何反應。
6. 回應量測（response measure）：怎樣判定符合標準。

範例：

| 欄位 | 內容 |
| --- | --- |
| 來源 | 一般學生 |
| 刺激 | 送出缺少必填欄位的報名表 |
| 環境 | 一般操作情境 |
| 對象 | 活動報名表單 |
| 回應 | 阻止送出、標示缺漏欄位、保留其他輸入 |
| 回應量測 | 每個缺漏欄位皆有繁體中文訊息，且已填內容不消失 |

這種寫法能讓「好用」轉成可以操作檢查的條件。

### 概念 8：業務規則、資料需求與限制條件不可混成一般功能

| 類型 | 核心問題 | 範例 |
| --- | --- | --- |
| 功能性需求（FR） | 系統必須做什麼？ | 系統應建立取消報名紀錄 |
| 非功能性需求（NFR） | 系統做到什麼程度？ | 取消完成後應立即顯示最新狀態 |
| 業務規則（business rule） | 組織規定什麼條件？ | 活動開始前 24 小時內不可自行取消 |
| 資料需求（data requirement） | 必須保存哪些資料？ | 取消時間、取消原因、操作者編號 |
| 限制條件（constraint） | 本專案受到什麼限制？ | 本期只使用假資料，不串接正式單一登入 |

分類的目的不是為了貼標籤，而是避免實作時遺漏判斷條件、資料欄位或專案限制。

### 概念 9：驗收條件定義「完成」

驗收條件（acceptance criterion）是判斷某一需求是否完成的具體條件。好的驗收條件應具備：

1. 對應明確需求編號。
2. 描述可觀察結果。
3. 一次檢查一個主要行為。
4. 包含必要的前提與資料狀態。
5. 不綁死不必要的技術實作方式。
6. 能由小組成員依相同步驟重現。

不夠好的寫法：

```text
報名功能正常。
```

較好的寫法：

```text
AC-05-01：活動剩餘名額大於 0，且學生尚未報名時，送出完整報名資料後，系統建立一筆報名紀錄並顯示「報名成功」。
```

### 概念 10：使用前提／當／那麼格式撰寫驗收條件

前提／當／那麼格式（Given/When/Then）把驗收條件分成三部分：

```text
前提（Given）：開始操作前的角色、資料與系統狀態。
當（When）：使用者執行的動作或系統發生的事件。
那麼（Then）：應觀察到的結果。
```

正常情境範例：

```text
AC-05-01
前提：活動尚有 3 個名額，學生尚未報名。
當：學生填完必填資料並送出報名。
那麼：系統建立報名紀錄、剩餘名額減少 1，並顯示「報名成功」。
```

例外情境範例：

```text
AC-05-02
前提：活動剩餘名額為 0。
當：學生嘗試送出報名。
那麼：系統不得建立正式報名紀錄，並顯示「名額已滿」。
```

邊界情境範例：

```text
AC-05-03
前提：活動只剩 1 個名額。
當：一名符合資格且尚未報名的學生完成報名。
那麼：系統建立報名紀錄，剩餘名額變為 0，後續畫面顯示「已額滿」。
```

### 概念 11：每個核心需求至少考慮三種情境

核心需求至少要檢查：

1. 正常情境（normal scenario）：資料正確、條件符合。
2. 例外情境（exception scenario）：資格不符、資料缺漏、操作不允許。
3. 邊界情境（boundary scenario）：名額 0 或 1、字數上限、時間截止點、重複操作。

不是每一則使用者故事都要寫十幾個條件，但不能只測最順利的情況。

### 概念 12：需求優先順序不是喜好排行

需求優先順序（requirements prioritization）應同時考量：

1. 使用者或組織價值（value）。
2. 時間急迫性（urgency）。
3. 法規、安全或隱私風險（risk）。
4. 是否為其他需求的前置依賴（dependency）。
5. 實作投入（effort）。
6. 本期範圍與展示價值（scope and demonstration value）。
7. 需求證據強度（evidence strength）。

「使用者說想要」不一定等於本次必須先做；「程式碼代理很容易產生」也不代表需求價值高。

### 概念 13：必須／應該／可以／本次不做方法

必須／應該／可以／本次不做方法（MoSCoW method）分為：

| 等級 | 英文 | 判斷方式 |
| --- | --- | --- |
| 必須有 | Must have | 沒有它就無法完成核心目標、違反必要規則，或無法展示主要流程 |
| 應該有 | Should have | 價值高，但短期缺少仍有暫時替代方式 |
| 可以有 | Could have | 有價值，但不影響本次核心流程與驗收 |
| 本次不做 | Won't have this time | 明確排除於本次範圍，避免默默膨脹 |

排序時必須寫理由。若所有需求都是「必須有」，表示小組沒有做出選擇。

建議限制：本次可操作切片中的「必須有」項目應維持在能於當日完成與驗收的數量。

### 概念 14：價值投入矩陣協助選擇實作順序

價值投入矩陣（value-effort matrix）用兩個面向判斷需求：

| 類別 | 處理方式 |
| --- | --- |
| 高價值、低投入 | 優先實作與驗收 |
| 高價值、高投入 | 拆小、先做可驗證切片 |
| 低價值、低投入 | 有餘裕再做，不因容易就自動加入 |
| 低價值、高投入 | 延後或移出本期範圍 |

投入估算可使用小／中／大，不要求精確工時，但要說明依據，例如畫面數量、資料狀態、外部服務、權限與例外情境。

### 概念 15：排序前要先檢查需求品質

品質不足的需求不應直接排序。先檢查：

1. 重複（duplicate）：兩筆需求是否描述同一件事。
2. 衝突（conflict）：兩筆需求是否要求相反行為。
3. 模糊（ambiguity）：同一句話是否可能有兩種解釋。
4. 遺漏（omission）：是否缺少角色、資料、例外或完成結果。
5. 不一致（inconsistency）：術語、狀態名稱或規則是否前後不同。
6. 不可驗收（not verifiable）：是否無法寫出觀察結果。
7. 越界（out of scope）：是否超出專案章程或本次切片。
8. 無來源（untraceable）：是否找不到需求證據。

發現衝突時，不可讓程式碼代理自行決定哪一個版本正確。應回到需求來源或提出待確認問題。

### 概念 16：需求文件包要讓下一個工作接得上

需求文件包（Requirements Package）不是把所有檔案放在一起而已。初稿至少應包含：

1. 專案與範圍摘要。
2. 利害關係人與角色。
3. 需求來源表。
4. 使用者故事清單。
5. 功能性需求清單。
6. 非功能性需求清單。
7. 業務規則與限制條件。
8. 驗收條件。
9. 需求優先順序。
10. 需求追溯矩陣初稿（requirements traceability matrix draft）。
11. 未解決問題清單（open issues list）。
12. 版本與變更紀錄（version and change log）。

文件包的完成標準是：另一位同學閱讀後，能知道這次做什麼、不做什麼、為什麼要做、先做哪個、如何驗收，以及還有哪些問題尚未確認。

### 概念 17：實作待辦清單把需求轉成可執行工作

實作待辦清單（implementation backlog）不是再抄一次需求。每個項目應包含：

1. 待辦編號。
2. 對應需求與驗收條件。
3. 要修改或新增的畫面、資料或行為。
4. 前置依賴。
5. 完成標準（Definition of Done, DoD）。
6. 驗證方式。
7. 優先順序。
8. 執行狀態。

範例：

```text
IMP-04：處理活動額滿狀態
對應：FR-05、AC-05-02、AC-05-03
工作：活動名額為 0 時停用正式報名，顯示額滿狀態；最後一個名額報名後更新畫面。
完成標準：三個指定測試情境都能重現，且沒有加入範圍外功能。
驗證：執行 MAT-05-02 與 MAT-05-03。
```

### 概念 18：第二個可操作切片要延伸核心流程

第二個可操作切片（second working slice）應延續第一個切片，形成更完整但仍可驗收的使用者流程。

適合的選擇方式：

1. 第一個切片完成「查看活動」，第二個切片完成「送出報名」。
2. 第一個切片完成「新增報修」，第二個切片完成「查看處理狀態」。
3. 第一個切片完成「提出申請」，第二個切片完成「承辦人核准或退回」。

不適合的選擇：

1. 尚未完成核心流程，就先製作報表或動畫。
2. 為了看起來完整而加入正式登入、付款或外部通知。
3. 一次加入多個角色、十個畫面與完整後台。

### 概念 19：更新實作任務書時要寫「變更」

更新後的程式碼代理實作任務書（updated code agent implementation brief）應明確列出：

1. 保留項目：既有功能中不能破壞的部分。
2. 新增項目：本次要加入的需求。
3. 修正項目：上次未符合驗收條件的地方。
4. 刪除項目：程式碼代理自行加入但不在範圍內的功能。
5. 不可變更項目：檔案結構、技術限制、介面語言或資料方式。
6. 驗收條件：本次完成後逐項檢查的標準。
7. 回報格式：完成檔案、操作方式、限制與未完成項目。

不要只寫「繼續完成系統」或「幫我修好」，因為程式碼代理無法知道小組目前認定的正確版本。

### 概念 20：需求與畫面對照表不是只有畫面名稱

需求與畫面對照表（requirements-to-screen mapping）應說明：

| 欄位 | 說明 |
| --- | --- |
| 畫面或操作編號 | 例如 `SCR-02` |
| 畫面或操作名稱 | 例如「活動詳情與報名」 |
| 對應需求 | 例如 `FR-05`、`NFR-03` |
| 對應驗收條件 | 例如 `AC-05-01` 至 `AC-05-03` |
| 可見元素 | 欄位、按鈕、狀態、提示訊息 |
| 資料變化 | 建立、修改、刪除或只讀 |
| 權限或條件 | 哪個角色、什麼狀態下可操作 |

如果畫面元素找不到對應需求，應先判斷它是否為必要的介面支援，或是未經確認的範圍外功能。

### 概念 21：手動驗收測試要留下證據

手動驗收測試（manual acceptance test）至少包含：

1. 測試編號。
2. 對應驗收條件。
3. 測試前置資料。
4. 操作步驟。
5. 預期結果。
6. 實際結果。
7. 通過或失敗。
8. 證據位置，例如截圖檔名或錄影時間點。
9. 問題與修正紀錄。
10. 修正後重測結果。

「我試過了，可以」不是驗收紀錄，因為其他人無法重現。

### 概念 22：程式碼代理可以協助實作，不能替小組批准需求

程式碼代理（code agent）可以協助：

1. 根據已確認的待辦項目修改程式碼。
2. 找出驗收條件可能對應的畫面與資料狀態。
3. 產生測試資料或手動測試步驟草稿。
4. 檢查檔案結構與實作影響範圍。
5. 說明修改內容與目前限制。

程式碼代理不能自行決定：

1. 新增未經需求來源支持的角色或功能。
2. 選擇衝突需求中的正確版本。
3. 把「本次不做」改成已實作範圍。
4. 宣稱需求已被利害關係人確認。
5. 以「程式可執行」取代人工驗收。

## 範例解析

以下範例沿用「校園社團活動報名系統」。所有內容是課堂練習範例，不代表各組必須採用相同規則或數值。

### 範例 1：從需求來源轉成使用者故事

原始需求來源：

```text
SRC-STU-03
來源角色：一般學生
來源摘要：學生希望在活動名額已滿時知道是否還能候補，以及目前候補順位。
來源強度：直接陳述
```

不佳轉換：

```text
系統要有候補功能。
```

較完整轉換：

```text
US-03
身為一般學生，
我想要在活動額滿時加入候補並查看目前順位，
以便判斷是否仍有機會參加活動。

來源：SRC-STU-03
```

轉換時沒有新增「自動寄信」「繳費」或「跨校候補」，因為來源未提供這些內容。

### 範例 2：從使用者故事拆成功能、規則與資料

使用者故事：

```text
US-03：身為一般學生，我想要在活動額滿時加入候補並查看目前順位，以便判斷是否仍有機會參加活動。
```

拆分結果：

| 類型 | 編號 | 內容 | 來源 |
| --- | --- | --- | --- |
| 功能性需求 | FR-07 | 當活動已額滿時，系統應允許符合資格且尚未報名的學生加入候補名單 | SRC-STU-03 |
| 功能性需求 | FR-08 | 系統應向候補學生顯示目前候補順位 | SRC-STU-03 |
| 業務規則 | BR-03 | 同一名學生在同一活動只能有一筆正式報名或候補紀錄 | SRC-STU-03、待確認 |
| 資料需求 | DR-04 | 候補紀錄應保存活動編號、學生編號、加入時間、候補順位與狀態 | 從 FR-07、FR-08 推導，待確認 |

注意：範例中的業務規則與資料欄位若不是直接來自來源，必須標示「推導、待確認」，不可偽裝成已訪談確認。

### 範例 3：建立可驗收的非功能性需求

模糊需求：

```text
系統要讓學生容易知道報名失敗的原因。
```

整理後：

```text
NFR-03：當報名未成功時，系統應在報名操作區附近顯示繁體中文訊息，說明失敗原因與下一個可採取動作，且不得清除使用者已填寫的其他有效資料。
```

可驗收原因：

1. 有事件：報名未成功。
2. 有位置：報名操作區附近。
3. 有內容：原因與下一個動作。
4. 有資料保留要求。
5. 可以透過操作觀察。

### 範例 4：為同一需求寫三種驗收條件

對應需求：

```text
FR-07：當活動已額滿時，系統應允許符合資格且尚未報名的學生加入候補名單。
```

正常情境：

```text
AC-07-01
前提：活動已額滿；學生符合資格；學生尚無報名或候補紀錄。
當：學生選擇「加入候補」。
那麼：系統建立候補紀錄，顯示「已加入候補」與候補順位。
```

例外情境：

```text
AC-07-02
前提：活動已額滿；學生已有候補紀錄。
當：學生再次嘗試加入候補。
那麼：系統不得新增第二筆紀錄，並顯示目前候補順位。
```

資格不符情境：

```text
AC-07-03
前提：活動已額滿；學生不符合活動參加資格。
當：學生查看活動詳情。
那麼：系統不得提供可執行的加入候補操作，並顯示不符合資格的原因。
```

### 範例 5：使用兩階段方法排序需求

第一階段使用必須／應該／可以／本次不做方法：

| 需求 | 分類 | 理由 |
| --- | --- | --- |
| FR-05 完成活動報名 | 必須有 | 核心目標，沒有它就無法展示主要流程 |
| FR-07 額滿時加入候補 | 應該有 | 能處理重要例外，但不影響基本報名流程成立 |
| FR-08 顯示候補順位 | 應該有 | 與候補流程直接相關，提升狀態透明度 |
| FR-12 匯出活動統計 | 可以有 | 對管理者有價值，但不影響學生報名流程 |
| FR-15 串接正式校務登入 | 本次不做 | 超出本期技術限制與資料權限範圍 |

第二階段比較價值與投入：

| 需求 | 價值 | 投入 | 依賴 | 建議順序 |
| --- | --- | --- | --- | --- |
| FR-05 | 高 | 中 | 活動資料 | 1 |
| FR-07 | 高 | 中 | FR-05、名額狀態 | 2 |
| FR-08 | 中 | 小 | FR-07 | 3 |
| FR-12 | 中 | 中 | 完整報名資料 | 延後 |
| FR-15 | 高 | 大 | 外部權限與正式服務 | 本次不做 |

### 範例 6：把排序結果轉成實作待辦清單

| 待辦編號 | 對應需求 | 工作內容 | 完成標準 | 優先順序 | 狀態 |

## 學生任務

今天所有成果必須以 7/13 的需求來源與專案範圍為依據。不得為了完成表格而捏造訪談結果，也不得讓程式碼代理自行增加需求。

### 任務 0：進入條件檢查

先確認下列項目：

| 檢查項目 | 完成 | 問題與補救方式 |
| --- | --- | --- |
| 專案章程第 2 版可開啟 |  |  |
| 需求來源表至少有 10 筆來源編號 |  |  |
| 候選需求清單已區分初步類型 |  |  |
| 7/13 實作任務書可開啟 |  |  |
| 第一個可操作切片可執行或有實作計畫 |  |  |
| GitHub 儲存庫連結可正常開啟 |  |  |

完成標準：

1. 六項均有檢查結果。
2. 未完成項目有補救方式與負責人。
3. 後續需求整理使用最新版本資料。

### 任務 1：建立使用者故事清單

從需求來源表選出本期最重要的需求，整理至少 8 則使用者故事（user stories）。

每則故事必須包含：

1. 使用者故事編號。
2. 角色。
3. 需求或目標。
4. 價值。
5. 來源編號。
6. 來源強度。
7. 是否需要進一步確認。

最低要求：

1. 至少 8 則使用者故事。
2. 至少涵蓋 2 種角色。
3. 每則故事至少對應 1 個需求來源編號。
4. 不得把生成式 AI 建議當成需求來源。
5. 每則故事都通過角色、需求與價值檢查。
6. 至少挑選 3 則故事使用六項品質原則進行檢查。

完成標準：另一組同學只看使用者故事，就能說出誰遇到什麼問題、希望完成什麼，以及為什麼重要。

### 任務 2：建立功能性需求清單

把使用者故事拆成至少 8 筆功能性需求（functional requirements）。

每筆功能性需求必須包含：

1. 需求編號。
2. 觸發角色或事件。
3. 前提條件。
4. 系統行為。
5. 可觀察結果。
6. 對應使用者故事。
7. 對應來源編號。
8. 待確認問題。

寫完後檢查：

```text
這句話是否描述系統行為，而不只是畫面名稱？
是否有明確觸發條件？
是否有可觀察結果？
是否超出專案章程範圍？
是否能寫出驗收條件？
```

完成標準：至少 8 筆功能性需求都有編號、來源與可觀察結果，且沒有把品質描述誤當成功能。

### 任務 3：建立非功能性需求清單

至少整理 5 筆非功能性需求（non-functional requirements），並涵蓋至少 3 種品質類型。

建議從下列類型選擇與專案相關者：

1. 效能（performance）。
2. 可用性（usability）。
3. 可靠性（reliability）。
4. 安全性（security）。
5. 隱私（privacy）。
6. 可維護性（maintainability）。
7. 相容性（compatibility）。
8. 可存取性（accessibility）。

每筆非功能性需求必須包含：

1. 編號。
2. 品質類型。
3. 適用情境。
4. 系統回應。
5. 可觀察或可測量標準。
6. 測試環境或限制。
7. 來源或推導依據。
8. 確認狀態。

完成標準：不得只寫「快速、安全、好用、穩定」；每筆需求都能說明如何驗收。自行暫定的數值須標示「待確認目標」。

### 任務 4：整理業務規則、資料需求與限制條件

從使用者故事與功能性需求中找出：

1. 至少 3 筆業務規則（business rules）。
2. 至少 3 筆資料需求（data requirements）。
3. 至少 3 筆限制條件（constraints）。

完成標準：每筆項目都有編號、內容、來源或推導依據；推導內容有標示「待確認」，限制條件與「本次不做」內容一致。

### 任務 5：撰寫驗收條件

為最高優先的使用者故事與功能性需求建立驗收條件（acceptance criteria）。

最低要求：

1. 至少 10 筆驗收條件。
2. 至少 5 筆使用前提／當／那麼格式。
3. 至少 3 筆正常情境。
4. 至少 2 筆例外情境。
5. 至少 2 筆邊界情境。
6. 至少 1 筆對應非功能性需求。
7. 每筆都對應需求編號。

完成標準：另一位同學能依條件準備資料、完成操作並判斷通過或失敗，不需要詢問「正常是什麼意思」。

### 任務 6：檢查需求品質與衝突

檢查全部需求，至少找出 3 個需要處理的品質問題。問題類型可包含：

1. 重複。
2. 衝突。
3. 模糊。
4. 遺漏。
5. 不一致。
6. 不可驗收。
7. 越界。
8. 無來源。

每個問題要記錄：

1. 問題編號。
2. 涉及的需求編號。
3. 問題類型。
4. 問題說明。
5. 修正方式或待確認問題。
6. 修正後版本。
7. 是否需要回到需求來源確認。

完成標準：至少 3 筆問題有具體修正；若無法判定，清楚列為未解決問題，不可讓程式碼代理猜答案。

### 任務 7：排定需求優先順序

先使用必須／應該／可以／本次不做方法（MoSCoW method），再使用價值投入矩陣（value-effort matrix）決定實作順序。

每筆需求至少要評估：

1. 分類。
2. 價值高、中、低。
3. 投入大、中、小。
4. 風險高、中、低。
5. 前置依賴。
6. 來源證據強度。
7. 排序理由。
8. 本次是否實作。

完成標準：

1. 至少有 1 筆「本次不做」項目。
2. 不得將所有需求都列為「必須有」。
3. 每筆高優先需求都有來源與驗收條件。
4. 排序結果符合當日可完成的實作範圍。

### 任務 8：組成需求文件包初稿

將前面成果整理成需求文件包初稿（Requirements Package draft）。

需求文件包至少包含：

1. 專案名稱、問題與範圍摘要。
2. 角色與利害關係人摘要。
3. 需求來源表連結。
4. 使用者故事清單。
5. 功能性需求清單。
6. 非功能性需求清單。
7. 業務規則、資料需求與限制條件。
8. 驗收條件。
9. 需求優先順序。
10. 需求追溯矩陣初稿。
11. 未解決問題。
12. 版本與變更紀錄。

完成標準：需求文件包中的編號一致，文件連結可開啟，沒有同一需求在不同檔案使用不同名稱或狀態。

### 任務 9：建立實作待辦清單

選擇本次要修正或新增的高優先需求，建立至少 4 筆實作待辦（implementation backlog items）。

每筆待辦包含：

1. 待辦編號。
2. 對應需求。
3. 對應驗收條件。
4. 工作內容。
5. 影響的畫面、資料或檔案。
6. 前置依賴。
7. 完成標準。
8. 驗證方式。
9. 優先順序。
10. 執行狀態。

完成標準：每筆待辦的範圍足以單獨理解，且能用對應驗收條件判斷完成。

### 任務 10：更新程式碼代理實作任務書

依需求優先順序與實作待辦清單，更新 7/13 的程式碼代理實作任務書。

必須寫清楚：

1. 目前系統狀態。
2. 本次目標。
3. 保留項目。
4. 修正項目。
5. 新增項目。
6. 刪除項目。
7. 本次不做。
8. 技術與資料限制。
9. 對應需求與驗收條件。
10. 執行順序。
11. 完成後回報格式。
12. 不可自行決定的事項。

完成標準：程式碼代理在不閱讀其他聊天紀錄的情況下，仍能知道要改什麼、不能改什麼、怎樣算完成。

共同要求：

2. 只處理實作待辦清單中的項目。

4. 不新增正式登入、金流、外部通知或正式資料庫等範圍外功能。
5. 介面文字使用繁體中文。
6. 保留程式碼代理的提示詞、回應摘要與人工修正。

完成標準：測試紀錄可由另一位同學重現；不得刪除失敗紀錄後只保留成功結果。

## 程式碼代理使用建議

程式碼代理（code agent）應分段使用。先要求檢查與提出計畫，確認沒有越界後再實作；實作後仍要由小組依驗收條件操作檢查。

### 提示詞 1：檢查需求來源轉換

```text
你是系統分析與設計課程中的需求檢查助手。

請檢查以下需求來源表與使用者故事，不要新增需求。

請逐項判斷：
1. 使用者故事是否有明確角色、需求與價值
2. 是否保留正確來源編號
3. 是否把來源中沒有的功能加進故事
4. 是否把解決方案誤寫成使用者需求
5. 哪些內容只是推論，必須標示待確認
6. 哪些故事太大，需要拆分

輸出格式：
- 故事編號
- 發現的問題
- 對應來源證據
- 建議修正
- 是否需要向利害關係人確認

不得宣稱任何需求已經被利害關係人確認。

[貼上需求來源表與使用者故事]
```

### 提示詞 2：檢查功能性與非功能性需求

```text
請檢查以下功能性需求、非功能性需求、業務規則、資料需求與限制條件。

檢查項目：
1. 類型是否正確
2. 功能性需求是否有觸發條件、系統行為與可觀察結果
3. 非功能性需求是否有情境、系統回應與可檢查標準
4. 是否有模糊詞，例如快速、好用、安全、智慧化、完整
5. 是否有重複、衝突、遺漏或範圍外需求
6. 每筆需求是否能追溯到使用者故事與來源
7. 自行推導的內容是否已標示待確認

請只指出問題與修正方式，不要自行創造新的利害關係人、規則、數值或需求來源。

[貼上需求清單]
```

```text
- 投入大、中、小

- 缺少的判斷資訊

若所有需求看起來都是必須有，請指出可以拆小、延後或移出本次範圍的項目。
```

## 常見錯誤與診斷

| 錯誤症狀 | 最可能原因 | 診斷問題 | 可驗證的修正步驟 |
| --- | --- | --- | --- |
| 使用者故事只有功能名稱 | 把系統方案當成使用者目標 | 是否有角色、目標與價值？ | 回到來源摘要，先寫使用者想完成的事情與原因 |
| 每則故事都寫「身為使用者」 | 角色分析未延續 | 不同角色的目標、權限與痛點是否相同？ | 對照利害關係人表，使用真實角色名稱 |
| 一則故事包含整套系統 | 沒有拆分可操作切片 | 是否包含多個動詞、角色或結果？ | 依操作流程或價值拆成多則故事 |
| 功能性需求只是畫面名稱 | 沒有描述系統行為 | 系統在什麼條件下做什麼？ | 補上觸發、行為與可觀察結果 |
| 非功能性需求寫成快速、安全、好用 | 缺少情境與量測方式 | 在什麼環境、對什麼行為、如何檢查？ | 使用品質屬性情境補齊回應與量測 |
| 非功能性需求數字看似精確但無來源 | 把暫定目標當成正式需求 | 數值由誰提出、如何量測？ | 標示待確認目標並記錄測試環境 |
| 業務規則被藏在驗收條件中 | 需求分類不完整 | 此條件是否影響多個需求？ | 抽成業務規則並讓相關需求共同引用 |
| 驗收條件只寫「功能正常」 | 沒有準備測試資料與結果 | 其他人能否依文字判斷通過？ | 改用前提／當／那麼格式 |
| 只寫正常情境 | 忽略例外與邊界 | 資料錯誤、名額為 0、重複操作時會怎樣？ | 每個核心需求至少補一項例外或邊界情境 |
| 驗收條件寫了介面顏色或框架 | 過早綁定技術方案 | 需求來源是否真的要求該技術細節？ | 改寫成使用者可觀察結果或移到設計文件 |
| 所有需求都是必須有 | 沒有範圍決策 | 移除哪一項，核心流程仍可成立？ | 使用價值、投入、依賴與證據強度重新排序 |
| 優先順序只看實作難度 | 忽略使用者價值 | 容易做是否等於重要？ | 先判斷價值與風險，再比較投入 |
| 程式碼代理加入新角色或功能 | 任務書範圍不清 | 是否列出本次不做與不可自行決定事項？ | 更新任務書並刪除未經確認的功能 |
| 程式碼代理一次修改太多檔案 | 待辦項目過大 | 能否指出每個修改對應哪項需求？ | 拆分待辦，每次只完成一組相關驗收條件 |
| 畫面可操作但找不到需求 | 缺少需求與畫面對照 | 每個按鈕、欄位、狀態來自哪裡？ | 建立對照表，刪除或說明無依據元素 |
| 測試全部標示通過但沒有證據 | 把自我感覺當成驗收 | 是否有步驟、實際結果與截圖？ | 重新執行並保存可重現紀錄 |
| 測試失敗後直接改成通過 | 沒有保留修正歷程 | 原始失敗與修正內容在哪裡？ | 保留失敗紀錄，建立待辦，修正後新增重測結果 |
| 文件與實作狀態不一致 | 修改後未同步版本 | 哪個版本是目前基準？ | 更新變更紀錄、追溯矩陣與提交紀錄 |
| 需求衝突時由程式碼代理選答案 | 把分析責任交給工具 | 衝突雙方的來源與權限是什麼？ | 停止該項實作，回到來源確認或列為未解決問題 |

## 形成性評量

### 檢核 1：概念辨識

下列哪一項是較完整的使用者故事？請說明原因。

```text
A. 身為學生，我想要報名功能。
B. 身為一般學生，我想要查看尚有名額且符合資格的活動，以便選擇並完成報名。
C. 系統要用資料庫保存活動。
```

參考答案：B。它包含明確角色、使用者目標與價值。A 只有功能名稱；C 是可能的技術或資料設計，不是使用者故事。

### 檢核 2：需求分類

請判斷類型：

| 內容 | 類型 |
| --- | --- |
| 系統應允許學生取消尚未開始的活動報名 |  |
| 報名失敗時應顯示失敗原因並保留已填資料 |  |
| 活動開始前 24 小時內不可自行取消 |  |
| 取消紀錄應保存取消時間與操作者 |  |
| 本期不串接正式校務登入 |  |

參考答案：

| 內容 | 類型 |
| --- | --- |
| 系統應允許學生取消尚未開始的活動報名 | 功能性需求（FR） |
| 報名失敗時應顯示失敗原因並保留已填資料 | 非功能性需求（NFR） |
| 活動開始前 24 小時內不可自行取消 | 業務規則（business rule） |
| 取消紀錄應保存取消時間與操作者 | 資料需求（data requirement） |
| 本期不串接正式校務登入 | 限制條件（constraint） |

### 檢核 3：改寫非功能性需求

請改寫：

```text
系統要好用。
```

可接受答案至少要有使用情境、系統回應與可檢查標準。例如：

```text
當使用者漏填必填欄位時，系統應在對應欄位附近顯示繁體中文錯誤訊息，指出缺漏內容，並保留其他已輸入資料。
```

### 檢核 4：改寫驗收條件

請把「取消報名功能正常」改成前提／當／那麼格式，至少寫一個正常情境與一個不允許取消的例外情境。

參考方向：

```text
正常情境
前提：學生已有報名紀錄，且目前時間仍在允許取消期間。
當：學生確認取消報名。
那麼：系統將報名狀態改為已取消，顯示取消成功，並更新可用名額。

例外情境
前提：學生已有報名紀錄，但已超過允許取消時間。
當：學生查看該筆報名。
那麼：系統不得提供可執行的取消操作，並顯示無法取消的原因。
```

### 檢核 5：需求排序判斷

某組把正式登入、活動瀏覽、活動報名、付款、聊天機器人與統計圖表全部列為「必須有」。請提出至少三個重新排序時應問的問題。

可接受答案可包含：

1. 哪一項直接支持專案章程中的核心問題？
2. 哪一項沒有完成就無法展示主要使用者流程？
3. 哪一項需要外部權限或正式資料，可能無法在本期完成？
4. 哪一項有明確需求來源與驗收條件？
5. 哪一項可先用較小切片驗證？
6. 哪一項即使容易實作，對使用者價值仍偏低？

### 檢核 6：程式碼代理判斷

程式碼代理認為加入電子郵件通知可以讓系統更完整，但需求來源、專案章程與本次待辦都沒有通知需求。應如何處理？

參考答案：不採用並記錄原因；不得因工具建議而直接加入。若小組認為可能有價值，應先列為待確認候選需求，取得需求來源後再進行排序與驗收設計。

### 檢核 7：追溯缺口判讀

請找出下列追溯資料的問題：

```text
SRC-02 → US-04 → FR-06 → AC-06-01 → SCR-03
SCR-04 → MAT-08-01
FR-09 → IMP-07 → SCR-05
```

參考答案：

1. 第一列尚未看到手動測試。
2. 第二列的畫面與測試沒有需求、來源與驗收條件。
3. 第三列沒有來源、使用者故事、驗收條件與測試。

### 檢核 8：離場檢查

離開前，每組必須能回答：

1. 今天最高優先的三項需求是什麼？為什麼？
2. 哪一項需求被列為本次不做？為什麼？
3. 哪一筆驗收條件曾經失敗？如何修正？
4. 本次實作修改了哪一段核心流程？
5. 哪些畫面或操作能追溯到需求與來源？
6. 還有哪個未解決問題不能交給程式碼代理自行判斷？
7. GitHub 上哪一個連結包含完整成果？

| 實作待辦與程式碼代理任務書 | 3 | 待辦可執行，任務書有保留、新增、修正、刪除、限制、驗收與回報格式 |

細部評分規準：

| 面向 | 4 分：完整且可追溯 | 3 分：達到主要要求 | 2 分：部分完成 | 1 分：需大幅修正 |
| --- | --- | --- | --- | --- |
| 需求品質 | 故事、功能、品質、規則、資料與限制分類清楚；每筆有來源與待確認狀態 | 多數需求可理解且有來源，少數分類或細節需修正 | 需求仍有模糊、混類或無來源項目 | 多數只有功能名稱，或內容由工具自行產生 |
| 驗收能力 | 正常、例外、邊界與品質條件可重現，能明確判斷通過或失敗 | 核心需求有驗收條件，少數前提或結果不完整 | 多數只測正常情境，或條件仍偏抽象 | 只寫「功能正常」或無法實際操作判斷 |
| 排序與範圍 | 優先順序有價值、投入、風險、依賴與證據依據，能明確取捨 | 有分類與理由，範圍大致可完成 | 多數項目未說明理由，或本次範圍仍過大 | 所有項目都是最高優先，沒有範圍決策 |
| 實作與代理使用 | 待辦、任務書、修改與驗收完全對應；保留人工決策與範圍控制 | 實作符合主要需求，紀錄大致完整 | 可操作但有越界、缺少對應或未記錄人工修正 | 直接要求代理完成整套系統，無需求控制 |
| 測試與追溯 | 來源到測試形成完整鏈；保留失敗、修正與重測證據 | 核心需求可追溯，多數測試有證據 | 追溯有多處缺口，失敗處理不完整 | 只有成功畫面，無測試步驟、證據或追溯 |

扣分與補交原則：

1. 捏造訪談內容、需求來源或確認狀態：該項不計分，並需補交來源與修正版。
2. 程式碼代理新增範圍外功能但未記錄：實作與代理使用項目最高 2 分。
3. 沒有失敗測試或重測紀錄：手動驗收項目不得評為完整。
4. GitHub 連結無法開啟或只上傳最終畫面：GitHub 與生成式 AI 紀錄項目不計分，修正後補評。
5. 需求編號前後不一致且無法追溯：需求文件包與追溯相關項目需補交。
